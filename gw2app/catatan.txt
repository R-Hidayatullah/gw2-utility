Ya. Dari dump struct yang kamu upload, alur file material Guild Wars 2 sudah cukup jelas.

## 1. AMAT container

```

 ├── GRMT chunk
 ├── DX9S chunk
 └── BGFX chunk
```

Terlihat pada hasil parser:

```

  GRMT (v6)
  BGFX (v1)
```

dan pada struct database juga ada:

```

```

# 2. Fungsi masing-masing

## MODL

Berisi mesh.

Setiap mesh memiliki

```
materialIndex
textures
constants[]
```

Jadi MODL membawa:

* texture override
* constant override
* material index

Bukan shader.

---

## GRMT

Berisi informasi material umum

```
texCount
texTokens[]
flags
sortOrder
```

Tidak ada shader bytecode.



---

## DX9S

Shader DirectX9 lama.

Di dalamnya ada

```
constRegisters[]
constTokens[]
shader bytecode
```

Ini pipeline lama.



---

## BGFX

Ini yang dipakai renderer modern.

Berisi:

* shader program
* sampler
* constants
* uniform

Dari reverse-engineering kamu:

```
sub_140BFB500
sub_140BFB060
```

memang membaca BGFX chunk, bukan DX9S.

# 5. Proses bind

Engine tidak memakai index.

Engine memakai token.

Algoritmanya kira-kira

```
for every BGFX constant
{
    token = shader.constant.token

    if MODL contains token
        upload MODL value
}
```

Jadi yang dicocokkan adalah

```
token/hash
```

bukan urutan.

---

# 6. Kenapa ada urutan?

Karena GPU constant buffer harus ditulis sesuai urutan shader.

Misalnya shader meminta

```
slot0 = token A
slot1 = token B
slot2 = token C
slot3 = token D
```

Maka engine membuat

```
cbuffer

slot0 = lookup(token A)

slot1 = lookup(token B)

slot2 = lookup(token C)

slot3 = lookup(token D)
```

Jadi:

```
shader order
        ↓
lookup hash
        ↓
ambil value MODL
        ↓
copy ke slot GPU
```

Urutannya mengikuti shader, bukan MODL.

---

# 7. Kenapa kamu dapat MISALIGNED?

Contoh

```
material-uniforms = 20

constantTokens = 29
```

Artinya shader mengharapkan

```
29 constant slot
```

tetapi AMAT hanya mengenal

```
20 material constant
```

Sisanya berasal dari

* object matrix
* world matrix
* camera
* lighting
* fog
* time
* shadow
* dll.

Jadi jelas tidak mungkin 20==29.

# 8. Rumus bind sebenarnya

Yang dilakukan engine kira-kira:

```cpp
for(slot=0; slot<constantCount; slot++)
{
    token = shader.constantTokens[slot];

    if(material.has(token))
        GPU[slot] = material[token];

    else if(renderer.has(token))
        GPU[slot] = renderer[token];

    else if(object.has(token))
        GPU[slot] = object[token];

    else
        GPU[slot] = default;
}
```

## Hash yang kamu lihat berasal dari mana?

Contohnya

```
0xeaac26b2
0x9e5874b0
0x4db5681e
```

**Itu bukan hash yang dibuat dari nama uniform shader.**

Itu adalah **ID/token yang sudah disimpan di file GRMT/AMAT**.

Lihat `sub_140BFB500`:

```cpp
v19 = *(_DWORD *)(v9 + *(_QWORD *)(a2 + 16));
```

Dia membaca token langsung dari data material.

Kemudian

```cpp
sub_140A67DD0(v19)
```

melakukan lookup:

```
token
    ↓
semantic id
```

Bukan

```
"specstr"
    ↓ hash ↓
0x9e5874b0
```

## Jadi fungsi `sub_140A67DD0` sebenarnya apa?

Dia hanya tabel lookup.

Misalnya

```
0x4DB5681E -> 0x5B
0x9E5874B0 -> 0x49
0xEAAC26B2 -> 0x70
```

Tidak ada algoritma hash di situ.

---

## Kapan hash dihitung?

Kemungkinan besar hanya:

* saat asset dibangun (ArenaNet asset compiler), atau
* saat GRMT dibuat.

Runtime game **tidak menghitung ulang**.

Ia tinggal membaca

```
GRMT

token
↓

lookup

↓

semantic

↓

UniformHandle
```

## Kalau membuat renderer sendiri

Yang paling sederhana bahkan tidak perlu semantic.

Kamu bisa langsung membuat map:

```cpp
std::unordered_map<uint32_t, bgfx::UniformHandle> materialBindings;
```

Misalnya

```cpp
materialBindings[0x9e5874b0] = u_specstr;
materialBindings[0xeaac26b2] = u_specpwr;
materialBindings[0x572d58f1] = u_mtlness;
...
```

Kemudian

```cpp
for (auto& c : amat.constants)
{
    auto it = materialBindings.find(c.token);

    if (it != materialBindings.end())
        bgfx::setUniform(it->second, c.value);
}
```

Tidak perlu menghitung hash sama sekali.

---

### Yang masih menjadi pertanyaan

Yang menarik justru **bagaimana `u_specstr` dipasangkan dengan token `0x9e5874b0`**.

Dari fungsi yang kamu kirim, saya belum melihat langkah itu secara eksplisit. `sub_140A67DD0()` hanya memetakan **token → semantic ID**, sedangkan `sub_140B65EC0()` mengambil informasi uniform BGFX. Masih ada satu langkah yang menghubungkan **semantic ID uniform shader** dengan **semantic ID dari token AMAT**.

Kalau kita menemukan fungsi itu, seluruh mekanisme binding material GW2 akan benar-benar lengkap tanpa perlu mengetahui algoritma hash sama sekali.
