# GW2 packfile parsing in minieditor (data-driven, no C++ recompile)

The engine understands GW2 packfiles from a **JSON registry** — you never
convert structs to C++/compile per format. To update formats after a game
patch: re-run `gen_gw2_json.py` in IDA and drop the new JSON into `templates/`.

## Files
- `templates/gw2_packfile.json` — ONE registry for all GW2 packfiles.
  Generated by `../gen_gw2_json.py` (IDAPython) from the IDB.
  Shape: `{ format, pointerSize, fileTypes{}, chunks{}, types{} }` — 28 file-types,
  69 unambiguous global chunks, 6582 struct types.

## How to use (GW2 Packfile panel)
1. Build: `cmake -S . -B build_new -G "MinGW Makefiles" && cmake --build build_new -t minieditor`
   (the old `build/` has a stale CMake cache pointing at a moved path).
2. Run `build_new/minieditor.exe` **from the `minieditor/` directory** so it finds `templates/`.
3. Drag a GW2 packfile (an extracted `PF...` blob) onto the window, or File > Open.
4. Go to the **"GW2 Packfile"** panel and click **"Detect PF (GW2)"**. It shows:
   - Header: container type, PF version, auto 32/64-bit, headerSize.
   - A table of every chunk in the file: fourcc | version | offset | size | resolved struct.
   - A **"Default file type"** combo — bulk-sets every row's strucTab to that file-type's.
   - A **"strucTab (pick)"** column — per chunk, a combo of every DISTINCT strucTab
     registered for that fourcc, labelled exactly like the output.txt header
     `strucTab 0x<addr>, versions:N [fileTypes]`. E.g. a `SKEL` chunk (whatever its
     file version) offers `strucTab 0x141EE4268, versions:2 [MODL,ROOT]` and
     `strucTab 0x142055260, versions:4 [PHYS]` — only one is correct for a given file;
     pick it per chunk without changing the others. If the chunk's version isn't present
     in a strucTab the option is tagged `(no vN)`. The `struct` column updates live.
   - Select a chunk row to preview its struct schema (wrapped, read-only) below.
   Registry keys: `fileTypes[container][chunk][ver]` (default resolution),
   `strucTabs[chunk][]` = `{tab,usedBy,versions}` (the pick list),
   and the UI sends `chunkOverrides[offset]→typeKey` to the engine on Parse.
5. Click **"Parse file into Tree"** — the full parse appears in the "Parsed Tree" panel;
   selecting a node highlights bytes in the hex view.

The "JSON Templates" panel is only for small hand-written linear templates
(e.g. `texture_example.json`). The GW2 registry is too large to edit there — it will
show a notice pointing you to the GW2 Packfile panel.

## 32 vs 64-bit — automatic
Detected from the PF header: pointer width = `(pfVersion & 4) ? 8 : 4` bytes
(matches Gw2 `Packfile.cpp`). PF v5 -> 64-bit, PF v1 -> 32-bit. No manual choice;
one template handles both. Override with top-level `"forcePointerSize": 32|64` if ever needed.

## File-type scoping (why one JSON, not "load everything")
The same chunk fourcc (e.g. `SKEL`, `ANIM`) is registered by several file-types
with DIFFERENT strucTabs. Resolving globally picks the wrong struct -> garbage.
So the registry is `fileTypes[containerFourcc][chunkFourcc][version] -> typeKey`.
The engine reads the container type at header offset 8 (e.g. `MODL`) and resolves
each chunk within that file-type's set; unambiguous chunks also live in a global
`chunks{}` fallback. On open, a `Header` node shows signature / pfVersion / container.

## Registry schema (what gen_gw2_json.py emits)
```
{ "format":"gw2packfile", "pointerSize":64,
  "chunks": { "MODL": { "70":"PackModelFileDataV70", ... }, ... },
  "types":  { "<key>": { "fields":[ <field>, ... ] }, ... } }
```
field.kind:
- scalars (fixed size): byte byte3 byte4 byte16 word word3 dword dword2 dword4
  qword float float2 float3 float4 double fileref token32 token64
- pointer-to-data (width=pointerSize): filename wchar_ptr char_ptr
- `struct`        + `type:"<key>"`                 (inline nested struct)
- `array`         + `element:<elem>` + `count:int` (inline fixed array)
- `array_ptr`     + `element:<elem>`               (u32 count + self-rel pointer)
- `ptr_array_ptr` + `element:<elem>`               (array of self-rel pointers)
- `ptr`           + `target:<elem>`                (single self-rel pointer)
`<elem>` = a scalar-kind string, or `{ "struct":"<key>" }`.

Pointers are **self-relative**: absolute = (byte offset of the pointer field) + stored value.

## Notes / limits
- Chunk routing matches the chunk fourcc literally against `chunks` keys. If a chunk
  shows "(no schema for this chunk/version)", verify its 4-char tag / version.
- Recursion is depth-capped (200) and node-capped (4M) to survive malformed input.
- `test_gw2.cpp` is a console harness that verifies pointer following on a synthetic
  buffer and smoke-loads the real registry:
  `g++ -std=c++20 -Iinclude -Iexternal/nlohmann-json/single_include test_gw2.cpp src/BinaryParser.cpp -o test_gw2 && ./test_gw2 templates/gw2_packfile.json`
